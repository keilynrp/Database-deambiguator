# Discovery Call Checklist

Checklist base para discovery comercial de UKIP antes de proponer demo o piloto.

Objetivo:

- entender el problema real antes de hablar de producto
- identificar stakeholder principal y audiencia secundaria
- detectar si existe una ruta clara hacia demo, piloto o propuesta

## 1. Preparación previa

- revisar el tipo de institución o equipo
- identificar hipótesis inicial del caso de uso
- definir cuál stakeholder parece más probable:
  - liderazgo institucional
  - oficina de investigación
  - biblioteca / metadatos
  - innovación / transferencia
- entrar con una hipótesis, no con una solución cerrada

## 2. Apertura de la conversación

Objetivo:

- entender el contexto sin ir demasiado rápido al pitch

Preguntas sugeridas:

- ¿Qué tipo de conversación interna los llevó a buscar algo como esto?
- ¿Qué les gustaría entender mejor sobre su portafolio o sus registros?
- ¿Qué está costando hoy más trabajo: confiar en los datos, leer el portafolio o comunicar hallazgos?

Señales a observar:

- dolor operativo
- dolor de confianza
- presión de reporte
- necesidad de priorización
- iniciativa estratégica en curso

## 3. Diagnóstico del problema

Objetivo:

- ubicar el problema principal en una de las capas donde UKIP aporta más valor

Preguntas sugeridas:

- ¿El problema principal hoy es calidad del dato, claridad analítica o narrativa para decisión?
- ¿Qué tan confiable sienten hoy la base sobre la que reportan?
- ¿Dónde suelen atascarse más: importación, armonización, validación, análisis o comunicación?

Mapeo rápido:

- si hablan de duplicados, inconsistencias, nombres ambiguos:
  - problema de confianza / metadata
- si hablan de no saber qué significa el portafolio:
  - problema de lectura analítica
- si hablan de no poder explicar readiness o gaps:
  - problema de narrativa ejecutiva

## 4. Stakeholder principal

Objetivo:

- descubrir para quién debe funcionar primero el piloto

Preguntas sugeridas:

- ¿Quién necesita sentir valor primero para que esto avance?
- ¿Quién usaría esto más seguido?
- ¿Quién tomaría la decisión de seguir después de un piloto?

Interpretación:

- liderazgo institucional:
  - quiere claridad, readiness y narrativa defendible
- oficina de investigación:
  - quiere cobertura, calidad, seguimiento y benchmark
- biblioteca / metadatos:
  - quiere confianza, authority y consistencia de registros
- innovación / transferencia:
  - quiere señal, outputs destacados y focos de seguimiento

## 5. Dataset y factibilidad

Objetivo:

- validar que existe material suficiente para una demo útil o un piloto acotado

Preguntas sugeridas:

- ¿Qué tipo de datos o registros tienen hoy disponibles?
- ¿Hay un subconjunto del portafolio que podamos usar como muestra inicial?
- ¿Existe alguna fuente prioritaria: CSV, repositorio, DOI list, ORCID, OpenAlex, etc.?

Señales de factibilidad:

- dataset pequeño o mediano disponible
- pregunta visible que el piloto debe responder
- disposición a revisar resultados con el equipo

## 6. Criterio de éxito

Objetivo:

- salir con una definición visible de éxito temprano

Preguntas sugeridas:

- Si esto funcionara bien en 4-6 semanas, ¿qué habría cambiado para ustedes?
- ¿Qué tendría que verse más claro o más confiable después del piloto?
- ¿Qué haría que digan “esto sí vale la pena seguirlo”?

Ejemplos de criterio visible:

- mayor cobertura de enrichment
- mejora en calidad/confianza del dataset
- lectura ejecutiva más defendible
- shortlist de prioridades más clara
- brief útil para conversación interna

## 7. Objeciones tempranas

Objetivo:

- detectar fricción antes de la propuesta

Preguntas sugeridas:

- ¿Qué les haría dudar antes de avanzar a una demo o piloto?
- ¿Hay alguna restricción de datos, tiempo o validación interna que debamos considerar?
- ¿Qué experiencias previas les hacen ser cautos con este tipo de herramientas?

Riesgos comunes:

- no hay dataset listo
- nadie dueño del piloto
- expectativa demasiado amplia
- buscan solución total inmediata
- presión por ver valor demasiado rápido

## 8. Cierre recomendado

Objetivo:

- no terminar con una conversación ambigua

Opciones de siguiente paso:

- demo enfocada por stakeholder
- piloto corto con muestra real
- propuesta de piloto
- follow-up con materiales específicos

Frase útil:

`Lo que más sentido haría desde aquí sería aterrizar un piloto corto, con una muestra real y un criterio visible de éxito, para validar si esta ruta realmente mejora la conversación que ustedes necesitan tener.`

## 9. Salida mínima de la llamada

Antes de cerrar, deberíamos poder responder:

- ¿cuál es el problema principal?
- ¿quién es el stakeholder principal?
- ¿qué dataset o muestra podría usarse?
- ¿cuál sería un criterio visible de éxito?
- ¿cuál es el siguiente paso concreto?

## 10. Artefactos relacionados

- stakeholder growth and sales strategy
- one-pagers por stakeholder
- pilot framing script
- demo script por stakeholder
- pilot proposal template
- objection handling sheet
- email follow-up template post-demo
