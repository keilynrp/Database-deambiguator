# Demo Script By Stakeholder

Guion base para demos de UKIP adaptadas por audiencia.

Objetivo:

- mostrar el mismo producto con una narrativa distinta segun el stakeholder
- reducir improvisacion en demos
- conectar discovery, producto y valor percibido

## 1. Estructura comun de demo

Toda demo deberia recorrer, en este orden:

1. contexto del problema
2. dataset real o tipo de dataset
3. flujo corto:
   - import
   - enrichment
   - review
   - dashboard
   - brief
4. lectura final para la audiencia
5. cierre con propuesta de piloto

## 2. Reglas generales

### Qué no hacer

- no abrir con jerga tecnica
- no abrir con arquitectura
- no abrir con menus complejos
- no vender “AI” o “knowledge graph” antes de explicar el problema

### Qué sí hacer

- abrir con una pregunta real de negocio
- usar un subconjunto de dataset
- mostrar una ruta corta y clara
- cerrar con un entregable entendible

## 3. Demo para liderazgo institucional / rectoria

### Apertura sugerida

`Lo que queremos mostrar aquí es si un portafolio institucional fragmentado puede convertirse en una lectura ejecutiva más clara y más defendible sin depender de semanas de preparación manual.`

### Qué enfatizar

- Executive Dashboard
- benchmark
- cobertura y calidad
- siguiente acción sugerida
- brief final

### Ruta recomendada

1. mostrar que el portafolio entra desde import
2. explicar que el enrichment fortalece confianza del dato
3. abrir el dashboard
4. leer readiness, quality y benchmark
5. abrir el brief con lente de liderazgo

### Frases útiles

- `Aquí no solo importa ver datos; importa saber si la historia ya es defendible.`
- `El benchmark no solo puntúa: también muestra qué frena la readiness.`
- `El brief final resume qué tan presentable es hoy esta lectura.`

### Resultado esperado

La audiencia debería salir con claridad sobre:

- qué tan madura está la lectura
- qué riesgo hay
- cuál es el siguiente movimiento

### Cierre sugerido

`Si esto se ve valioso, el siguiente paso sería correr el mismo flujo con un subconjunto real de su portafolio y validar si ya sostiene una conversación ejecutiva defendible.`

## 4. Demo para oficina de investigación

### Apertura sugerida

`Lo que queremos mostrar aquí no es solo un dashboard, sino cómo pasar de un dataset disperso a una base más confiable para seguimiento y reporte institucional.`

### Qué enfatizar

- import y scientific import
- enrichment coverage
- quality
- authority review
- benchmark rules

### Ruta recomendada

1. mostrar entrada del dataset
2. explicar enrichment y mejora de señal
3. mostrar tabla de conocimiento
4. diferenciar estado del sistema, estado de revisión y calidad
5. abrir dashboard y benchmark
6. mostrar brief con lente de oficina de investigación

### Frases útiles

- `La pregunta no es solo si podemos reportar, sino si podemos confiar en la base del reporte.`
- `Aquí la mejora visible suele aparecer en cobertura, calidad y claridad de gaps.`
- `El benchmark ayuda a traducir esas señales a una lectura operativa.`

### Resultado esperado

La audiencia debería salir con claridad sobre:

- qué ya es usable
- qué todavía necesita atención
- dónde conviene enfocar el siguiente esfuerzo operativo

### Cierre sugerido

`Si esto aplica a su contexto, el piloto natural sería medir cómo mejora la confianza del dataset y qué tan reutilizable queda como línea base de seguimiento.`

## 5. Demo para biblioteca / metadatos

### Apertura sugerida

`Aquí el foco no es solo analítica. El foco es si podemos mejorar la confianza en la capa de registros para que luego analítica y reporte descansen sobre una base más estable.`

### Qué enfatizar

- authority review
- harmonization
- trazabilidad
- tabla de conocimiento
- brief con lente de biblioteca / metadatos

### Ruta recomendada

1. mostrar registros con ruido o variantes
2. abrir authority review
3. explicar revisión, calidad y sistema
4. mostrar harmonization log
5. cerrar con la lectura del brief para esta audiencia

### Frases útiles

- `Sin confianza en la capa de registros, el resto de la lectura se debilita.`
- `Aquí la mejora no se ve solo en una métrica: se ve en la consistencia del dataset.`
- `El producto ayuda a distinguir progreso técnico de revisión humana.`

### Resultado esperado

La audiencia debería salir con claridad sobre:

- qué tan confiable es hoy la capa de registros
- dónde sigue habiendo ambigüedad
- dónde conviene invertir curación

### Cierre sugerido

`Si esto resuena, el piloto adecuado sería trabajar un lote real de registros ambiguos y medir la mejora en confianza y consistencia del dataset.`

## 6. Demo para innovación / transferencia

### Apertura sugerida

`Aquí no buscamos mostrar todo el portafolio; buscamos ver si el dataset ya produce una lectura rápida de outputs destacados y áreas de concentración que valga la pena seguir.`

### Qué enfatizar

- top entities by impact
- knowledge concept map
- dashboard ejecutivo
- brief con lente de innovación / transferencia

### Ruta recomendada

1. mostrar que el dataset entra y se enriquece
2. ir al dashboard
3. leer top entities y concept map
4. conectar eso con oportunidad o exploración
5. cerrar con el brief para esta audiencia

### Frases útiles

- `La pregunta aquí es si el portafolio ya deja de ser ruido y empieza a producir orientación.`
- `No buscamos profundidad total desde el día uno; buscamos señal útil para decidir dónde mirar.`
- `El valor aparece cuando outputs y concentraciones ya justifican seguimiento.`

### Resultado esperado

La audiencia debería salir con claridad sobre:

- qué outputs destacan
- qué concentración temática emerge
- cuál sería el siguiente movimiento exploratorio

### Cierre sugerido

`Si esto encaja, el piloto natural sería evaluar si un subconjunto real del portafolio ya produce señales útiles para oportunidad, vinculación o seguimiento.`

## 7. Cierre comun de demo

### Fórmula sugerida

`Lo importante de esta demo no es que vean todas las funciones, sino que podamos validar si, con un subconjunto real de su portafolio, UKIP ya mejora la claridad de la conversación que ustedes necesitan tener.`
