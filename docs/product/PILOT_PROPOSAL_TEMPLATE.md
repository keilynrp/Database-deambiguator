# Pilot Proposal Template

Plantilla base para proponer un piloto UKIP a un prospecto o stakeholder interno.

## 1. Título

`Pilot Proposal — UKIP`

Subtítulo sugerido:

`Validating whether a real portfolio can become a clearer and more defendable decision-ready narrative`

## 2. Contexto

### Situación actual

[Nombre de la organización o equipo] necesita entender mejor un subconjunto real de su portafolio, pero hoy la lectura está limitada por:

- fragmentación del dataset
- baja confianza en parte de los registros
- poca claridad sobre qué señales son suficientemente confiables para compartir o decidir

### Audiencia principal

- [liderazgo institucional / oficina de investigación / biblioteca / innovación]

### Pregunta central del piloto

`¿Puede UKIP convertir este subconjunto real del portafolio en una lectura más confiable, útil y accionable para [stakeholder]?`

## 3. Objetivo del piloto

El objetivo del piloto es validar si, con un dataset acotado y real, UKIP puede:

- importar y estructurar el portafolio
- enriquecer y fortalecer la señal del dato
- visibilizar calidad, gaps y readiness
- producir un dashboard y un brief útiles para la audiencia principal

## 4. Alcance

### Dataset inicial

- tipo de dataset: [publicaciones / autores / afiliaciones / mixto]
- volumen inicial: [ej. 500–1,500 registros]
- dominio principal: [science / health / humanities / otro]

### Stakeholder principal

- [perfil principal]

### Entregables del piloto

- workspace funcional con dataset importado
- lectura de cobertura, calidad y benchmark
- revisión inicial de autoridad/calidad cuando aplique
- dashboard ejecutivo
- brief ajustado al stakeholder principal

## 5. Flujo de trabajo del piloto

1. seleccionar y recibir dataset inicial
2. importar y mapear datos
3. ejecutar enrichment
4. revisar calidad y autoridad
5. analizar dashboard ejecutivo
6. generar brief final para la audiencia acordada

## 6. Criterios de éxito

El piloto se considerará exitoso si demuestra al menos una de estas condiciones:

### A. Narrativa ejecutiva más clara

- el brief ya sostiene una lectura defendible
- o deja explícito qué frena esa lectura

### B. Mejora operativa visible

- mejora cobertura, calidad o claridad de gaps
- el dataset queda en una base más reutilizable

### C. Señal exploratoria útil

- emergen outputs, clústeres o señales que justifican seguimiento

## 7. Qué mediremos

- total de entidades o registros cargados
- cobertura de enrichment
- calidad promedio
- benchmark readiness
- principales gaps o restricciones
- claridad del brief para el stakeholder principal

## 8. Supuestos y límites

### Supuestos

- el prospecto comparte un subconjunto real y usable del portafolio
- existe una audiencia principal claramente definida
- el éxito del piloto se medirá sobre una pregunta concreta

### Límites

- este piloto no busca desplegar toda la plataforma
- no busca resolver todos los problemas de calidad del dataset
- no busca cerrar de inmediato un flujo enterprise completo

## 9. Resultado esperado

Al final del piloto, [nombre del equipo/organización] debería poder responder con mayor confianza:

`[pregunta clave del piloto]`

Y contar con:

- una lectura más clara del estado real del portafolio
- una narrativa más útil para [stakeholder]
- una base concreta para decidir el siguiente paso

## 10. Siguiente paso después del piloto

Si el piloto valida valor, las rutas naturales de expansión son:

- ampliar volumen de dataset
- institucionalizar benchmark
- repetir el brief de forma recurrente
- abrir workflows/reportes o una segunda fase de producto

## 11. Cierre sugerido

`La propuesta de este piloto no es probar todas las capacidades de UKIP, sino validar con un caso real si el producto ya mejora la claridad de la conversación que ustedes necesitan tener. Si el resultado no mejora esa conversación, lo sabremos rápido. Si sí la mejora, ya tendremos una base concreta para decidir cómo escalar.`
